# E10-S1 — Preview URLs for Draft Releases

**Epic:** Epic 10 — Review, Automation, and Lifecycle
**Status:** Implemented (2026-03-03)
**Priority:** P1
**Estimated Effort:** 4-5 days
**Dependencies:** E4-S1 (release history), E5-S2 (Caddy config generation), E5-S3 (safe reload), E9-S2 (multi-route Caddy generation)
**Target:** `internal/db/`, `internal/server/`, `internal/caddy/`, `internal/client/`, `internal/cli/`
**Design Reference:** PRD 10.4 preview URLs roadmap item

---

## 1. Summary

Add first-class preview URLs that bind a non-active release to an expiring hostname under an operator-configured preview base domain. This lets operators and reviewers inspect a specific release artifact without activating it in staging or prod and without rebuilding any content.

## 2. Architecture Context and Reuse Guidance

- Reuse the existing immutable release model. A preview points at an existing `releases/<id>/` directory; it never mutates or rebuilds the release.
- Reuse current Caddy generation instead of adding a second HTTP stack. Preview hosts are just extra Caddy site blocks with roots that target a specific release directory.
- Reuse the telemetry retention cleanup loop pattern for preview-expiry cleanup. Do not invent a new goroutine lifecycle mechanism.
- Keep previews separate from `domain_bindings`. Domains map hosts to environment `current`; previews map hosts to one specific release.
- Keep preview configuration server-side:
  - `preview.enabled`
  - `preview.baseDomain`
  - `preview.defaultTTLHours`
  - `preview.maxTTLHours`
- Use `internal/domain.Normalize` for `baseDomain` validation.

## 3. Proposed Changes and Architecture Improvements

### 3.1 Preview model

Add a `release_previews` table keyed by environment + release + hostname:

- `id INTEGER PRIMARY KEY`
- `environment_id INTEGER NOT NULL`
- `release_id TEXT NOT NULL`
- `hostname TEXT NOT NULL UNIQUE`
- `created_by TEXT NOT NULL DEFAULT ''`
- `expires_at TEXT NOT NULL`
- `created_at TEXT NOT NULL`

Indexes:

- `idx_release_previews_env`
- `idx_release_previews_expires_at`

The hostname must be generated by the server, not accepted from the client. Use a random lowercase token plus normalized website/environment labels:

`<token>--<env>--<website>.<previewBaseDomain>`

This avoids path/domain injection and keeps the CLI surface small.

### 3.2 Caddy generation

Extend `internal/caddy.Site` with optional headers so preview sites can emit:

- `X-Robots-Tag: noindex, nofollow, noarchive`

Preview site blocks otherwise behave like normal sites:

- root points at the pinned release directory
- backends remain environment-scoped and are loaded for the preview environment
- telemetry proxying remains active when telemetry is enabled

### 3.3 API and CLI

Endpoints:

- `POST /api/v1/websites/{website}/environments/{env}/previews`
- `GET /api/v1/websites/{website}/environments/{env}/previews`
- `DELETE /api/v1/websites/{website}/environments/{env}/previews/{id}`

CLI:

- `htmlctl preview create website/<name> --env staging --release <id> [--ttl 72h]`
- `htmlctl preview list website/<name> --env staging`
- `htmlctl preview remove website/<name> --env staging --id <preview-id>`

`create` returns the generated hostname and expiry time.

### 3.4 Cleanup and retention interaction

- Expired previews must be deleted automatically by a cleanup loop and on server startup.
- Preview cleanup triggers Caddy regeneration only when at least one preview was removed.
- This story establishes a release pin that later retention/GC work must preserve.

## 5. Implementation Steps

1. Add migration 008 and query helpers for preview CRUD and expiry cleanup.
2. Extend server config with preview settings and validate:
   - `enabled=false` permits empty preview config
   - `enabled=true` requires valid `baseDomain`
   - TTL defaults must be positive and `defaultTTLHours <= maxTTLHours`
3. Implement preview hostname generation in the server. Do not trust client-supplied hostnames.
4. Implement `handlePreviews` with create/list/delete behavior and auth enforcement.
5. Extend Caddy generation to include preview host site blocks with `X-Robots-Tag`.
6. Add preview cleanup loop modeled on telemetry retention cleanup.
7. Add CLI commands and structured output.
8. Document the operator prerequisite: wildcard DNS for `*.previewBaseDomain` must resolve to the server.

## 6. Tests and Validation

### Automated

- DB tests:
  - insert/list/delete preview rows
  - delete expired previews
- Server tests:
  - create preview for known release returns `201`
  - invalid TTL returns `400`
  - missing release returns `404`
  - unauthenticated requests return `401`
  - delete unknown preview returns `404`
  - 5xx responses are sanitized
- Caddy tests:
  - preview host block points at release directory, not `current`
  - preview host emits `X-Robots-Tag`
  - preview host keeps telemetry and backend routing
- CLI tests:
  - create/list/remove call correct endpoints
  - table/json output is deterministic

### Manual

- Apply a release, create preview, open returned hostname, verify content matches the pinned release.
- Promote or roll back the environment and confirm the preview still serves the pinned release.
- Wait for expiry or force cleanup and verify the hostname stops resolving after Caddy reload.

## 8. Risks and Open Questions

### Risks

- **Preview hostname becomes mistaken for access control.**
  Mitigation: document clearly that previews are routable hostnames, not auth. AuthPolicy is a separate story.
- **Preview reload churn from many expirations.**
  Mitigation: cleanup runs on a coarse interval and reloads only when rows were actually deleted.
- **Preview storage pins old releases.**
  Mitigation: later retention work must treat preview rows as retention pins.

### Open Questions

- None blocking. v1 uses one server-wide preview base domain, not per-website preview domains.
